## Deliverable Summary
Complete project architecture for adding MCP server to FinaryExport solution:

1. **Project structure:**
   - Extract `FinaryExport.Core` shared library (zero using-statement changes via RootNamespace=FinaryExport)
   - New `FinaryExport.Mcp` console app (RootNamespace=FinaryExport.Mcp)
   - CLI retains current `.csproj` structure

2. **MCP tools (15 total across 7 classes):**
   - UserTools: `finary_get_org_context`, `finary_get_user_profile`
   - PortfolioTools: `finary_get_portfolio_summary`
   - AccountTools: `finary_get_all_accounts`, `finary_get_accounts_by_category`, `finary_get_account_timeseries`
   - TransactionTools: `finary_get_all_transactions`, `finary_get_transactions_by_category`, `finary_get_transaction_categories`
   - DividendTools: `finary_get_all_dividends`
   - HoldingsTools: `finary_get_all_holdings`
   - AllocationTools: `finary_get_portfolio_allocations`, `finary_get_allocation_distribution`, `finary_get_allocation_fees`

3. **Auth strategy:**
   - Per user directive: MCP reuses `session.dat` from CLI
   - No cold auth / no env var credentials / no Otp.NET
   - `McpCredentialPrompt` throws if no session exists

4. **Transport & dependencies:**
   - Stdio transport (standard for VS Code Copilot, Claude Desktop)
   - SDK: `ModelContextProtocol` 1.1.0 (official)
   - Shared auth/API via `FinaryExport.Core`

5. **Implementation notes:**
   - All 15 tools are read-only (GET)
   - Bootstrap required: `finary_get_org_context` before any data tool
   - Transaction category restriction: 4 of 10 categories support transactions
   - Per-category error isolation for aggregation tools
   - Rate limiter already in place (FinaryDelegatingHandler)

## Key Decisions
- Core library extraction keeps "FinaryExport" namespace — zero cascade impact
- Session-only auth per user directive — simplifies MCP, removes Otp.NET
- All read-only operations (no mutations exposed)
- Phase 1: expose all 15 methods; future phases could add `PUT /users/me/ui_configuration` if needed
